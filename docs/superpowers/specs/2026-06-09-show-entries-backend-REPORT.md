## 1. Реализовано по спеке (готово к интеграции)

### Собака теперь знает своего владельца

- В `Dog` появился `owner_id` — пользователь, добавивший собаку.
- **`DogResponse` теперь содержит `owner_id: uuid | null`** (есть везде, где
  отдаётся собака: `GET /dogs`, `GET /dogs/{id}`, `POST /dogs`, `GET /users/me/dogs`).
- Семантика для UI: прятать кнопки «изменить»/«удалить»/«записать» у чужих
  собак — `dog.owner_id === currentUser.id`.
- `owner_id === null` — легаси-собака без сопоставленного владельца
  (см. бэкафилл ниже). Трактуй как «не моя».

### Новый эндпоинт «мои собаки»

```
GET /users/me/dogs          (auth: Bearer, обязателен)
  query: page=1 (>=1), per_page=50 (1..200)
  -> 200 DogPage { items: DogResponse[], total, page, per_page }
```

- Возвращает только собак текущего пользователя (`owner_id == me`),
  **включая собак без питомника**.
- Сортировка: `name asc`. Дополнительных фильтров (search/breed) пока нет —
  по спеке фронту не требовались.
- Формат страницы — та же плоская обёртка, что у `GET /dogs`.

### Бэкафилл (что видно у существующих пользователей)

- Старым собакам владелец проставлен из питомника
  (`dog.owner_id = kennel.owner_id`).
- На текущих данных: 58 собак, **50 получили владельца**, 8 легаси-собак
  без питомника остались `owner_id = null` (исторически владелец неизвестен).
- Вывод для UI: собаки в питомниках пользователя видны в `/users/me/dogs`
  сразу. «Ничьи» бесхозные собаки — нет (это ожидаемо).
- Собаки, созданные **после** правки, всегда получают `owner_id = создатель`.

### Запись на выставку — защита от чужой собаки

```
POST /shows/{show_id}/entries   (auth)
  body: ShowEntryCreate { dog_id, show_class_id, handler_id?, notes? }
  -> 201 ShowEntryResponse
```

- Теперь **нельзя записать чужую собаку**: разрешено владельцу
  (`dog.owner_id == me`), либо admin, либо (фолбэк для легаси) владельцу
  питомника собаки. Иначе `403 forbidden`.
- Контракт не менялся — тело и ответ те же.
- Бизнес-ошибки приходят `4xx` с машиночитаемым `detail` (показывай тостом):
  `registration_not_open`, `registration_deadline_passed`,
  `dog_already_registered`, `class_not_available_for_age`,
  `class_animal_type_mismatch`, `breed_not_allowed`, `dog_birth_date_missing`,
  `forbidden`, `*_not_found`.

### Без изменений (подтверждено, работает)

```
GET /shows/{show_id}/entries/my                      -> ShowEntryResponse[]
GET /shows/{show_id}/available-classes/{dog_id}      -> AvailableClassesResponse
```

## 2. ⚠️ Сопутствующие изменения контракта (НЕ из этой спеки, но в том же мерже)

Эти правки тоже попали в `main` и **меняют формат ответов** — фронту нужно
адаптироваться, иначе соответствующие экраны сломаются.

### `GET /kennels` — теперь страница, а не массив

Было: `KennelResponse[]` (голый массив).
Стало: `KennelPage { items: KennelResponse[], total, page, per_page }`.

Списки питомников нужно читать из `response.items`, total брать из `total`.

### `GET /breeds` — плоская пагинация вместо вложенного `meta`

Было: `{ items, meta: { total, page, per_page } }`.
Стало: `{ items, total, page, per_page }` (как DogPage/KennelPage).

`PageMeta` удалён. Если фронт читал `response.meta.total` — поправить на
`response.total`.

### `DELETE /dogs/{dog_id}/images/{file_id}` — открепление фото (новый)

Удаляет связь фото с собакой (файл в хранилище остаётся). Право: владелец
питомника собаки или admin; собака без питомника — только admin. При
откреплении главного фото главным становится фото с наименьшим `position`.
Возвращает обновлённый `DogResponse`. (Реализация запроса из
`2026-06-08-dog-image-delete.md`.)

## 3. Чек-лист интеграции для фронта

- [ ] Экран «Мои собаки» / форма записи — переключить на `GET /users/me/dogs`.
- [ ] Использовать `dog.owner_id` для скрытия действий у чужих собак.
- [ ] Запись на выставку — обрабатывать `403 forbidden` (чужая собака) и
      прочие `4xx detail` тостами.
- [ ] **Список питомников** — читать `items`/`total` из `KennelPage`.
- [ ] **Список пород** — `response.total` вместо `response.meta.total`.
- [ ] (опц.) Кнопка открепления фото у собаки — `DELETE /dogs/{id}/images/{file_id}`.

Контракт сверять с `:8000/openapi.json` — он уже отражает новый маршрут,
поле `owner_id` и изменённые обёртки пагинации.
