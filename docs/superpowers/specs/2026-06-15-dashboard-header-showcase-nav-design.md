# Дизайн: публичная навигация в хедере дашборда

Дата: 2026-06-15
Статус: согласование

## Проблема

Сайдбар дашборда перегружен (~14 пунктов). Публичные страницы платформы —
питомники (`/kennels`), животные (`/animals`), выставки (`/shows`) — должны быть
доступны из **навигации хедера**, а сайдбар дашборда остаётся под управление и
админку. Сейчас хедер дашборда не содержит навигации по публичным разделам.

Это первый шаг большей работы. Следующие шаги (НЕ входят в этот дизайн):
вынести фильтры (статус/город/поиск) на публичный `/shows` и убрать дублирующую
админ-таблицу `/dashboard/shows`.

## Область этого шага

Добавить в хедер дашборда ту же публичную навигацию, что на публичном сайте:
**Главная · Питомники · Животные · Выставки** → `/`, `/kennels`, `/animals`,
`/shows`.

Вне области: изменение сайдбара, фильтры публичного `/shows`, удаление таблицы
дашборда, перенос пунктов из сайдбара.

## Текущее устройство (отправная точка)

- Публичный сайт: `MainLayout` (`src/layouts/main/layout.tsx`) рендерит
  `NavDesktop` с `mainNavData(t)` (`src/layouts/nav-config-main.tsx`) в правой
  части хедера; на мобиле — `NavMobile` (showcase-вариант) в бургер-дровере.
- `mainNavData` — плоский список пунктов `{ title, path, icon }` типа
  `NavMainProps['data']`: Главная, Питомники, Животные, Выставки.
- Дашборд: `DashboardLayout` (`src/layouts/dashboard/layout.tsx`). В режиме
  `vertical` хедер тонкий: слева — кнопка-бургер (мобайл), справа — язык /
  уведомления / настройки / аккаунт. Навигации по публичным разделам нет.
- Мобильный дровер дашборда — `src/layouts/dashboard/nav-mobile.tsx`
  (`NavSectionVertical`, данные `NavSectionProps['data']` — секции). Делит
  `navData` с desktop-сайдбаром `NavVertical`.

## Решение

Переиспользовать существующие компоненты, без нового кастома.

### Desktop

В `DashboardLayout` импортировать `NavDesktop` и `navData as mainNavData` и
вставить горизонтальное меню в `leftArea` хедера, после кнопки-бургера,
видимое только на `>= layoutQuery` (desktop):

```
leftArea:
  <MenuButton .../>           // как сейчас, только мобайл
  <NavMobile ... />           // дровер дашборда (см. ниже)
  <NavDesktop                 // НОВОЕ: публичная навигация
    data={mainNavData(t)}
    sx={{ display: 'none', [up(layoutQuery)]: { display: 'flex' } }}
  />
  // (логотип/divider только для horizontal-режима — без изменений)
```

`NavDesktop` уже умеет ровно это (горизонтальный ряд пунктов с активным
состоянием по `usePathname`). Источник правды — единый `mainNavData`, тот же,
что на публичном сайте.

### Mobile

У дашборда нет хедер-меню — его роль на мобиле играет бургер-дровер. Добавляем
публичные ссылки **отдельной секцией только в дровер**, не затрагивая
desktop-сайдбар.

`NavVertical` (desktop-сайдбар) продолжает получать прежний `navData`.
`NavMobile` (дровер) получает `[showcaseSection, ...navData]`, где

```
showcaseSection = {
  subheader: t('navbar:main.menu')   // напр. «Сайт» / «Showcase»
  items: mainNavData(t)              // { title, path, icon } совместимы с NavSection
}
```

Формы данных совместимы: пункты `mainNavData` — `{ title, path, icon }`, что
принимает `NavSectionVertical`. Фильтрация прав к этой секции не применяется
(публичные ссылки видны всем) — пункты без поля `permission` проходят
`filterNavItems` без ограничений.

### i18n

Нужен один новый ключ подзаголовка секции для мобильного дровера, напр.
`navbar:main.menu` (RU «Сайт», EN «Site») — добавить в `src/locales` (RU+EN).
Сами пункты (`main.home/kennels/animals/shows`) уже переведены.

## Затрагиваемые файлы

- `src/layouts/dashboard/layout.tsx` — импорт `NavDesktop` + `mainNavData`;
  вставка desktop-меню в `leftArea`; сборка `showcaseSection` и передача
  расширенных данных в `NavMobile` (сайдбар — без изменений).
- `src/locales/langs/{ru,en}/navbar.json` (или эквивалент) — ключ `main.menu`.

Без изменений: `nav-config-main.tsx`, `nav-config-dashboard.tsx`,
`NavVertical`, `NavHorizontal`, `ShowListView`, `ShowShowcaseView`.

## Краевые случаи

- **Режим `horizontal` дашборда.** Сейчас в этом режиме в хедер уже уезжает
  дашбордная навигация (`NavHorizontal` в `bottomArea`) + логотип/divider в
  `leftArea`. Публичный `NavDesktop` в `leftArea` будет соседствовать с
  логотипом — приемлемо, но проверить визуально, чтобы не было тесноты. При
  необходимости показывать публичный `NavDesktop` только когда
  `!isNavHorizontal`. По умолчанию режим `vertical`, поэтому это редкий путь.
- **Плотность хедера на узком desktop.** Лого нет в vertical-режиме, слева
  место есть; правый блок компактный. 4 пункта помещаются. Контролируется
  `layoutQuery` (меню скрыто ниже брейкпоинта, доступно в дровере).
- **Активное состояние.** `NavDesktop`/`NavList` подсвечивают активный пункт по
  `usePathname` — на публичных роутах подсветка работает; внутри `/dashboard/*`
  ни один публичный пункт не активен (ожидаемо).

## Проверки

- `npx tsc --noEmit` → 0 ошибок.
- `npm run lint` → 0 (импорты через `eslint --fix`).
- `npm test` → зелёно (логика не меняется; новых юнитов не требуется).
- Ручная проверка: в дашборде desktop виден ряд Главная/Питомники/Животные/
  Выставки, клики ведут на `/`, `/kennels`, `/animals`, `/shows`; на мобиле те
  же ссылки есть в бургер-дровере отдельной секцией; desktop-сайдбар без
  публичных пунктов.

## Следующие шаги (вне этого дизайна)

1. Перенести фильтры (статус/город/поиск/ранг) на публичный `/shows`
   (`ShowShowcaseView`).
2. Убрать дублирующую админ-таблицу `/dashboard/shows` (`ShowListView`) после
   того, как публичная страница покроет сценарий, оставив таблицу/CRUD только
   там, где это реально админка.
3. Почистить сайдбар дашборда от пунктов, уехавших в публичную навигацию.
